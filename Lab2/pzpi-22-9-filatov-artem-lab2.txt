1.1 Мета завдання:
Метою цієї лабораторної роботи є створення мобільного програмного
застосунку на платформі Android із використанням мови програмування
Kotlin. Додаток є частиною комплексної системи моніторингу якості
середовища в офісних приміщеннях, яка також включає IoT-пристрій,
API-сервер і веб-інтерфейс (що розроблятиметься пізніше). Застосунок
повинен надавати користувачу можливість переглядати поточні параметри
повітря, температуру, вологість тощо, а також отримувати сповіщення про
відхилення від встановлених норм. Реалізація має демонструвати
архітектурну інтеграцію з серверною частиною та чітку структуризацію
функціоналу.
1.2 Хід роботи:
1. Було проаналізовано вимоги до мобільного застосунку як частини
загальної системи моніторингу параметрів середовища.
2. Обрано технологічний стек: Kotlin + Android Studio для клієнтської
частини, REST API для комунікації із сервером.
3. Сформульовано основні функціональні можливості мобільного
додатку: авторизація, перегляд поточних показників, історія змін,
отримання сповіщень при відхиленнях, налаштування сповіщень.
4. Створено UML-діаграму прецедентів (Use Case Diagram) для
візуалізації взаємодії користувача із застосунком.
5. Побудовано діаграму компонентів (Component Diagram) для
відображення структури застосунку та його зв’язку з іншими
елементами системи.
6. Додатково створено діаграми станів і діяльності для моделювання
поведінки застосунку.
7. Розроблено початкову структуру мобільного застосунку в Android
Studio та налаштовано базову взаємодію з API (емульований сервер).
8. Проведено тестування основних сценаріїв: вхід у систему, запит
поточних даних, обробка сповіщень.
2. UML діаграма прецедентів (Use Case Diagram);
Ця діаграма демонструє функціональність мобільного застосунку з позиції
користувача. Вона описує основні дії користувача, такі як перегляд
параметрів повітря, отримання сповіщень та взаємодія із сервером через
API.
3. UML діаграмa компонентів (Component Diagram).
Діаграма відображає архітектуру мобільного додатку як сукупність
компонентів: UI, ViewModel, Repository та API-клієнта. Вона також
ілюструє взаємодію з віддаленим сервером.
4. UML Activity Diagram — Діаграма діяльності
Описує послідовність дій користувача після запуску мобільного застосунку
до перегляду даних та отримання сповіщення при перевищенні норм.
5. UML State Diagram — Діаграма станів
Показує стани, в яких може знаходитися мобільний застосунок:
очікування, отримання даних, показ даних, помилка.
6. ER-модель мобільної платформи
Ця ER-модель відображає структуру основних таблиць, що зберігаються у
централізованій базі даних PostgreSQL, яка обслуговує мобільний
застосунок через REST API, усі дані постійно зберігаються та
оновлюються на сервері.
Висновок:
У ході лабораторної роботи було реалізовано архітектурну та
функціональну основу мобільного застосунку для системи моніторингу
якості параметрів середовища в офісних приміщеннях. Було прийнято
обґрунтовані інженерні рішення щодо технологічного стеку та структури
взаємодії з API. Додаток вже на поточному етапі забезпечує базову
взаємодію з системою та має потенціал для подальшого масштабування
функціональності, зокрема за рахунок інтеграції з ШІ, додавання нових
типів сенсорів або підтримки офлайн-режиму. Робота підтверджує
доцільність обраного підходу до архітектури мобільного застосунку в
рамках загальної системи.
